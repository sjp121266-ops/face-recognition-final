# 技术设计说明

## 设计目标

本项目的目标是完成一个可演示、可解释、可写报告的安卓人脸识别期末作业。系统选择本地离线识别，而不是云端 API，主要原因是课程演示更需要稳定性，云端方案容易受账号、网络、额度和接口权限影响。本机演示模式不上传人脸图片或特征，模型推理和人脸库匹配都在设备本地完成。

在本地离线主线稳定后，项目新增云端 API 模式。云端模式支持 Face++ 托管 API 和 CompreFace 自部署 REST 服务。这样系统可以在答辩中展示两套识别引擎：本地 FaceNet TFLite 引擎用于断网稳定演示，云端引擎用于展示线上服务化架构。

## 模块职责

- `MainActivity`：组织 App 主流程，包括权限、相机回调、页面切换、状态展示和记录展示；本地识别、云端识别和多人视频帧策略已下沉到协调类，避免主类继续膨胀。
- `CameraController`：封装 CameraX 预览、拍照、视频帧分析、前置摄像头优先绑定、后置摄像头回退和全屏预览切换，避免相机绑定细节继续堆在 Activity 中。
- `LocalRecognitionCoordinator`：协调本地录入和识别，负责同名补录、最多 5 组样本保留、离群样本拒绝和识别结果转换。
- `CloudRecognitionCoordinator`：协调云端录入、云端识别和连接测试，把 Activity 与 Face++ / CompreFace 路由细节隔开。
- `LiveFrameCoordinator`：协调多人视频帧中的人脸几何质量检查、跨帧稳定结果转换和视频演示记录门槛判断，并把轨迹、投票和 L2 距离等调试信息作为报告材料保存。
- `CloudRecognitionRouter`：根据当前云端提供商选择 Face++ 或 CompreFace 网关，避免云端分发逻辑继续堆在 Activity 中。
- `CloudFaceGateway`：定义云端连接测试、录入和识别的统一接口。
- `CloudFaceClient`：调用 CompreFace REST API，完成云端录入和云端识别。
- `FacePlusPlusClient`：调用 Face++ 托管云端 API，完成 FaceSet 初始化、云端录入和云端搜索识别。
- `CloudFaceSettings`：保存云端提供商、服务地址、API Key、API Secret、FaceSet outer_id 和云端相似度阈值；当前保存方式仅服务课程演示，正式产品应改为服务端代理或加密密钥管理。
- `CloudConnectionStatusFormatter`：格式化最近一次云端连接测试状态，展示测试时间、提供商和成功/失败原因。
- `BitmapUtils`：处理拍照结果，将 `ImageProxy` 转成 `Bitmap`，裁剪人脸区域，并为云端演示生成 JPEG 字节。
- `FaceNetModel`：加载 TensorFlow Lite 模型并输出人脸特征向量。
- `LiveRecognitionStabilizer`：对视频多人识别结果做跨帧跟踪、短窗口投票和平滑显示，避免单帧波动导致标签跳动或多人串标。
- `EnrollmentAdvisor`：根据某个人员已录入的特征组数给出补录建议，帮助提高识别稳定性。
- `FaceEmbeddingGuard`：检查人脸特征向量维度和数值有效性，计算 L2 距离，并在同名补录时拦截离群样本。
- `RecognitionEngine`：使用综合 L2 距离、候选人间隔和样本稳定性完成身份匹配；内部只接受 128 维有限向量，会过滤维度错误、NaN、Infinity 等异常向量，并在判定解释中展示跳过数量。
- `FaceLibraryHealthAnalyzer`：检查人脸库是否具备演示条件，校验 128 维特征向量，评估同人样本一致性、跨人员区分度和阈值校准建议，并清理异常数据。
- `FaceQualityAnalyzer`：在特征提取前检查人脸大小和角度，减少低质量录入带来的识别不稳定。
- `FaceImageQualityAnalyzer`：检查裁剪后人脸图像的亮度和基础清晰度，阻止过暗、过曝或明显模糊的样本进入模型。
- `RecognitionRecordManager`：创建识别记录、插入到列表顶部，并裁剪到最近 30 条，避免记录管理规则堆在 Activity 中。
- `ProfileRepository`：包装本地人脸库内存状态和持久化写入，集中处理替换、更新、删除人员和清空人脸库，避免 Activity 直接调用存储细节。
- `RecordRepository`：包装识别记录内存状态和持久化写入，集中处理新增记录、上限裁剪和清空记录。
- `DemoStateSnapshot`：为 UI 刷新提供轻量状态快照，集中表达当前人员、记录、本地/云端模式和实时识别状态，减少后续继续拆 UI 时的状态散落。
- `RecognitionRecordFormatter`：格式化识别记录，避免记录展示逻辑堆在 Activity 中。
- `DemoCoverageAnalyzer`：根据识别记录判断答辩验收场景是否覆盖，并生成下一步补充建议。
- `DemoGuideBuilder`：根据当前人脸库和识别记录生成首页“下一步”提示，帮助演示者按正确顺序完成录入、识别、未知人员、多人视频和摘要生成。
- `TestSummaryBuilder`：根据本地人脸库和识别记录生成测试摘要，支持界面展示和复制到剪贴板，方便截图或粘贴到报告。
- `AlgorithmEvaluationBuilder`：根据人脸库和识别记录生成算法样本评测说明，解释同人 L2、跨人 L2、未知人员 L2、风险控制信号和阈值建议。
- `AdvancedReadinessBuilder`：生成长期算法与安全增强边界，说明活体检测、模型升级、性能 delegate 和正式发布前各自需要的证据，避免把未验证加分项误写成已完成能力。
- `EvidenceChecklistBuilder`：生成真机证据链清单，提醒哪些验收项已有记录可整理，哪些仍必须用真实设备截图或录屏补证据。
- `CloudDemoMaterialBuilder`：生成云端 API 演示材料，说明 Face++、CompreFace、本地离线和 GPT 多模态模型的取舍。
- `FullReportBuilder`：生成可复制到报告的完整素材，包含截图摘要、技术详情、人脸库明细、算法样本评测、真机证据链、完整识别记录、云端 API 演示材料、隐私与风险说明和报告使用建议。
- `FaceStore`：底层 JSON 文件读写和旧版 SharedPreferences 迁移，读取失败时自动兜底；上层通过 Repository 使用它。

## 数据流

录入流程：

1. 用户输入姓名。
2. CameraX 拍照。
3. ML Kit 检测单张人脸。
4. 检查人脸大小、角度和画面边缘距离。
5. 裁剪人脸区域。
6. 检查裁剪后人脸图像亮度和基础清晰度。
7. FaceNet 提取 128 维向量。
8. 如果是同名人员补录，会与已有样本比较；差距过大则拒绝加入，避免脏样本污染人脸库。
9. 保存到本地人脸库。
10. 根据当前人员特征组数提示是否建议继续补录。

识别流程：

1. CameraX 拍照。
2. ML Kit 检测单张人脸。
3. 检查人脸大小、角度和画面边缘距离。
4. 裁剪人脸区域并检查亮度和基础清晰度。
5. 送入 FaceNet，提取当前人脸向量。
6. 与本地人脸库逐个计算 L2 距离。
7. 计算每个人的综合 L2 分数：`score = minDistance + (averageDistance - minDistance) * 0.35`。
8. 取综合分数最低的人员作为第一候选，同时记录第二候选距离。
9. 如果分数低于阈值 `10.0` 且第一/第二候选差距足够大，显示候选姓名；否则显示未知人员或拒绝确认，并展示原因。
10. 保存识别记录。

视频多人识别流程：

1. CameraX ImageAnalysis 从预览流中按固定间隔抽取帧。
2. ML Kit 检测当前帧中的多张人脸。
3. 对每张人脸分别做人脸质量检查、裁剪和 FaceNet 特征提取。
4. 对每张人脸调用同一个 `RecognitionEngine`。
5. `LiveRecognitionStabilizer` 使用人脸框位置匹配前后帧轨迹，并对同一轨迹的最近几帧身份结果做投票；短暂误判不会立刻替换稳定姓名，只有新身份获得足够票数后才切换显示。每个轨迹在同一帧最多匹配一张脸，降低多人串标风险。
6. 将每张脸的人脸框、姓名/未知标签传给 `FaceOverlayView`，叠加显示在相机预览上。
7. 在结果区按“第 1 张、第 2 张……”列出姓名、相似度、距离或未知原因。
8. 稳定器会为每张可稳定化人脸附带轨迹 ID、匹配方式、投票数、确认阈值和 L2 距离等调试信息；这些信息默认不污染画面标签，只写入视频演示记录和完整报告，便于解释跨帧确认。
9. 视频模式只有在至少两张人脸同时入镜、且出现有效稳定身份标签后，才写入一条“视频演示”记录，用于答辩验收覆盖；该记录只写一次，避免连续帧把记录刷满。

云端 API 识别流程：

1. 用户在首页切换到“云端 API”模式。
2. 用户选择 Face++ 托管云端或 CompreFace 自部署。
3. Face++ 填写 API Key、API Secret 和 FaceSet outer_id；CompreFace 填写服务地址和 Recognition API Key。
4. 用户可先点击“测试云端连接”：Face++ 会创建/确认 FaceSet，CompreFace 会调用 `GET /api/v1/recognition/subjects/` 验证服务地址和 API Key。
5. App 保存最近一次连接测试状态，包含测试时间、提供商、成功/失败和说明；若切换到另一个云端提供商，界面会提示重新测试。
6. App 仍然使用 CameraX 拍照和 ML Kit 检测单张人脸，保证上传给云端的是质量合格的人脸图像。
7. 云端录入时，Face++ 会 Detect 得到 `face_token`、设置 `user_id` 并加入 FaceSet；CompreFace 会调用 `/api/v1/recognition/faces?subject=姓名`。
8. 云端识别时，Face++ 调用 Search API；CompreFace 调用 `/api/v1/recognition/recognize`。
9. App 根据云端返回的最高相似度/置信度和默认阈值 `0.75` 显示“云端识别成功”或“云端未匹配”。

云端模式不替代本地离线模式，而是作为对比增强：本地模式强调离线稳定和隐私，云端模式强调服务化和多端共享人脸库。

## 稳定性设计

- 模型加载失败时禁用录入和识别按钮，并显示原因。
- 本地 JSON 数据损坏时自动清空对应数据，不让 App 崩溃。
- 本地人脸库加载后会校验特征向量维度和数值有效性，移除非 128 维、NaN、Infinity 等异常特征和无有效特征的人员，避免坏数据导致识别崩溃。
- 录入新样本时，`FaceEmbeddingGuard` 会先确认模型输出是 128 维有限向量；同名补录时会计算新样本与已有有效样本的最近 L2 距离，距离过大时拒绝加入，避免把错误人员或极差质量样本混入人脸库。
- 测试摘要固定展示“本机 TFLite FaceNet 模型 + 本机 JSON 人脸库；识别流程不调用云 API”，作为断网演示和报告截图的离线证明。
- 识别引擎内部还有二次防御：当前人脸特征不是 128 维或包含异常值时直接拒识，本地库中的异常向量会被跳过，并在识别说明中展示跳过数量，避免绕过存储清理后触发崩溃。
- 同一人员有多组样本时，会计算组内最大 L2 距离；如果样本差异超过默认识别阈值，系统会提示该人员样本不稳定，建议重新补录。
- 多个人员同时存在时，会计算最近跨人员 L2 距离；如果两个人距离过近，系统会提示区分度偏低，建议补录更清晰的正脸样本。
- 当同人样本和跨人样本之间存在清晰距离间隔时，系统会给出建议阈值；如果两者重叠，则提示不要盲目调高阈值，应先重新补录。
- 未检测到人脸、多张人脸、裁剪失败、拍照失败、推理失败都有中文提示。
- 人脸太小、太靠近画面边缘、侧转角度过大、头部倾斜过大、画面偏暗、过曝或明显模糊时会阻止录入/识别，并提示用户调整姿势。
- 首页会显示人脸库健康和演示准备度；测试摘要会进一步输出样本一致性、跨人区分度和阈值校准建议，帮助答辩前快速确认数据是否准备好。
- 首页会显示“下一步”提示：空库时提示录入人员；人员不足时提示录入第二位测试人员；样本不足时提示补录；记录缺失时依次提示补充成功识别、未知人员和多人视频演示；覆盖完成后提示生成截图摘要。
- 测试摘要会分析识别记录覆盖情况，提示识别成功、未知人员、边界/拒识、候选人过近和视频多人识别等验收场景是否已经准备好。
- 测试摘要和完整报告会生成算法样本评测说明，帮助用真机样本解释默认 L2 阈值、同人一致性、跨人区分度和风险控制信号。
- 完整报告会生成真机证据链清单，把安装启动、录入识别、未知人员、异常场景、多人视频、断网识别、小屏截图和云端对比拆成可核对证据项；清单不会替代真实设备验证。
- 云端连接测试成功和失败都会保存到界面状态中，避免答辩前忘记当前 API Key、FaceSet 或自部署服务是否可用。
- 人脸库和识别记录分开清空，避免误删全部数据。
- 每人最多保留 5 组特征向量；识别记录由 `RecognitionRecordManager` 统一写入和裁剪，最多保留 30 条。
- 单人录入、单人识别、云端上传和多人视频识别共用带轻微扩边的人脸裁剪策略，减少人脸框过紧造成的特征波动。
- 视频多人识别每隔固定时间分析一帧，减少实时推理压力，避免低端设备卡顿；识别标签会经过跨帧投票和平滑显示，只有跨帧确认后的多人结果才写入答辩验收记录。记录中会保留紧凑调试摘要，例如轨迹号、匹配方式、票数和 L2 距离，方便排查跳标或串标。
- 云端错误会按 Key/权限、服务地址、额度限制、超时、服务端异常等常见原因转成中文提示；云端仍是加分对比模块，不影响本地离线主线。
- 删除人员、清空人脸库、清空识别记录都需要二次确认，减少演示时误触风险。

## UI 与动效设计

本项目的 UI 目标不是做复杂页面，而是让答辩流程在手机上更容易被老师看懂。主页面保留录入、拍照识别和全屏多人视频三个核心动作，云端配置、识别记录、报告素材和数据管理收纳到“设置与材料”页，避免主演示流程被低频设置打断。

原生动效由 `MotionDirector` 集中管理，它只接收 View 并执行入场、按钮反馈、状态切换、页面切换和全屏进出场动画，不持有人脸库、识别结果或云端配置等业务状态。动效优先使用 `alpha`、`translationX/Y` 和 `scaleX/Y`，并尊重系统关闭动画设置。

动效和视觉反馈主要服务四个场景：

1. 首页入场：打开 App 后优先呈现相机预览、人脸库状态、演示准备度、2x2 演示流程卡、下一步提示和主要操作按钮，让演示者知道下一步该做什么。
2. 状态切换：录入成功、识别成功、未知人员、质量不足和候选人过近等结果需要在同一区域清楚更新，避免旧结果残留造成误读。
3. 全屏多人识别：视频模式强调铺满屏幕的相机预览、人脸框、姓名/未知标签和扫描反馈，帮助观众直观看到系统正在逐张分析画面中的人脸。
4. 小屏适配：小屏或窄屏设备上，长文案、按钮、标签和底部结果区必须保持可读可点；当前版本会在 360dp 左右窄屏上收紧首页标题、材料按钮、三枚核心操作按钮和全屏多人视频底部面板，减少结果区遮挡关键人脸的概率。如果真机发现重叠，应记录设备型号和截图，作为后续兼容优化依据。

当前终端环境只能确认文档中的验收标准和构建检查已经补齐，不能替代真机视觉验收。首页入场、状态切换、全屏多人识别扫描效果和小屏不重叠需要在答辩前用真实设备截图或录屏确认。

## 隐私与风险说明

- 本地演示模式下，人脸特征向量和识别记录保存在 App 本机私有存储中，识别流程不调用云 API。
- 当前 APK 已关闭 Android 自动备份，降低人脸特征、识别记录和云端演示配置被系统备份带走的风险。
- Debug 包保留明文 HTTP 访问能力，方便课程演示连接 `10.0.2.2` 或局域网 CompreFace；Release 包通过构建占位符关闭全局明文 HTTP，并开启资源压缩和代码混淆，避免把 Debug 边界误当成正式发布边界。
- 云端对比模式只有在用户主动切换并填写 API 信息后才会上传裁剪后的人脸图片，用于课程演示中的云端人脸库录入和搜索。
- 云端 API Key/API Secret 当前仅为课程演示临时保存在本机配置中，正式系统不应把密钥保存在客户端。
- 人脸图片和特征属于敏感生物特征数据，课程演示应征得测试人员同意，不采集无关人员，不用于商业、门禁、金融核身等高风险场景。
- 当前项目未实现活体检测，无法完全防止照片翻拍攻击，因此报告中应将活体检测列为后续安全增强。
- 第四阶段新增的是可解释边界和检查能力，不是直接接入活体模型、替换 FaceNet 或启用 GPU/NNAPI；这些改动必须先完成真机 A/B 数据、阈值重标定、误识/拒识统计和性能记录。

## 测试设计

`RecognitionEngine` 是身份匹配的核心逻辑，不依赖 Android 相机、ML Kit 或 TensorFlow Lite，因此适合做本地单元测试。当前测试覆盖：

- 空人脸库。
- 录入建议测试覆盖样本不足和达到推荐数量两种情况。
- 精确向量匹配。
- 多人员最近平均距离匹配。
- 多样本综合 L2 分数匹配。
- 超过阈值返回未知人员，并保留最近候选人用于解释阈值判断。
- 第一候选和第二候选过近时拒绝确认。
- 匹配结果包含安全区、稳定区、边界区和拒识区解释。
- 人员没有特征数据。
- 当前人脸特征不是 128 维、库中特征维度错误或包含 NaN/Infinity 时不会崩溃，而是返回未知人员或跳过异常样本。
- 人脸特征守卫测试覆盖向量维度、NaN/Infinity、L2 距离、同名补录离群拒绝和无有效历史样本时不误拒。
- 人脸几何质量评估测试覆盖合格人脸、人脸过小、贴边、侧转过大、倾斜过大。
- 人脸图像质量评估测试覆盖偏暗、过曝、模糊和亮度清晰度合格。
- 人脸库健康测试覆盖空库、样本不足、可演示状态、异常向量清理、同人样本一致性风险、跨人员混淆风险和阈值建议。
- 视频识别稳定器测试覆盖连续帧确认、单帧噪声保持稳定结果、新身份赢得投票后切换、过期轨迹清理和同帧轨迹不复用。
- 视频识别调试信息测试覆盖新轨迹 ID、匹配类型、投票数、确认阈值、同帧轨迹不复用和调试摘要写入视频演示记录。
- 识别记录管理测试覆盖新记录插入顶部、超过上限时删除最旧记录、手动裁剪保留最新记录。
- 识别记录格式化测试覆盖空记录、普通记录、无距离记录。
- 验收覆盖分析测试覆盖空记录、完整覆盖和部分覆盖建议。
- 演示引导测试覆盖空库、人员不足、样本不足、成功识别、未知人员、多人视频和云端模式连接测试提示。
- 测试摘要生成逻辑覆盖空数据和混合识别结果统计。
- 云端路由测试覆盖 Face++ 与 CompreFace 提供商分发，确保连接测试、录入和识别会交给当前选中的云端网关。
- 云端演示材料测试覆盖 Face++、CompreFace、GPT 多模态模型取舍、云端记录统计和云端连接测试步骤。
- 完整报告测试覆盖摘要、人脸库明细、完整识别记录、云端 API 演示材料、隐私与风险说明和报告使用建议是否同时生成。
- 算法样本评测和真机证据链测试覆盖空数据、样本不足、距离统计、风险信号和多人视频证据项。
- 长期算法与安全增强测试覆盖证据不足时不建议接入活体、样本清晰分离时允许进入模型 A/B 评测准备、只有存在多人视频记录后才建议评估 NNAPI/GPU/XNNPACK delegate。
- 云端连接状态格式化测试覆盖尚未测试、连接成功、连接失败和切换提供商后需重新测试等提示。
- Android 仪器测试覆盖 `FaceStore` 本机存储读写、`BitmapUtils` 裁剪工具和 `MainActivity` 启动后核心答辩按钮可见；这些测试可以构建 Android Test APK，但仍不替代真实摄像头、人脸和小屏截图验收。

这些测试证明阈值判断和候选选择规则可单独验证，便于答辩时说明项目不是只依赖界面演示。

## 为什么云端识别作为第二模式

云端识别通常需要服务部署、API Key 和网络请求。对于期末作业来说，本地离线识别仍然是最稳定的主线；云端 API 模式作为第二模式，用于展示服务化架构、REST API 调用和云端人脸库管理能力。演示时即使云端服务不可用，本地离线模式仍可完成核心验收。

## 为什么不做活体检测

活体检测属于安全增强，不是身份识别的最低要求。静态活体或动作活体需要额外模型或平台接口，也会增加误判风险。本项目在报告中将其列为后续改进，保证主功能稳定完成。
