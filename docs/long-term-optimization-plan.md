# 人脸识别期末作业长期优化计划书

## 目标

这个项目的长期目标不是堆功能，而是持续把它打磨成一个“能演示、能解释、能扩展”的安卓人脸识别作品。每次迭代都应该服务三个结果：演示更稳定、用户更容易理解、报告更有内容可写。

## 当前版本定位

当前版本是本地离线识别演示台：

- CameraX 负责拍照和预览。
- ML Kit 负责人脸检测。
- FaceNet TFLite 负责提取 128 维特征向量。
- L2 距离、候选人间隔和样本稳定性负责本地身份匹配。
- CameraX ImageAnalysis 支持视频流多人脸识别。
- 应用私有目录 JSON 文件保存人员特征和识别记录，并兼容旧版 SharedPreferences 数据迁移。

这个版本适合作为期末答辩主版本，因为它不依赖网络、云账号或后端服务。

## 迭代原则

- 优先保证主流程：录入、识别、记录、异常提示必须始终可用。
- 每个新增功能都要能在答辩里讲清楚价值。
- 不引入会显著增加演示失败风险的依赖。
- 危险操作必须有确认，结果必须有解释。
- 文档和代码同步更新，避免报告与实际 App 不一致。

## 阶段 1：演示可靠性

目标：让 App 在普通安卓手机上稳定完成 3 分钟演示。

计划：

- 保留拍照式识别作为稳定主流程，同时提供视频多人识别作为展示能力。
- 所有失败路径都显示中文提示。
- 对删除人员、清空人脸库、清空记录增加二次确认。
- 让空人脸库、空识别记录、空人员选择都有明确空状态。
- 每次构建后运行 `./gradlew assembleDebug --no-daemon`。

验收：

- 录入 2 人，每人 2 次。
- 识别已录入人员成功。
- 未录入人员显示未知人员。
- 视频多人识别可以在预览画面中逐张输出多个人脸身份。
- 断网后仍可识别。
- 误点删除类按钮不会立刻清空数据。

## 阶段 2：用户体验

目标：让第一次使用的人也知道下一步该做什么。

计划：

- 把主界面维持为“准备状态 -> 主操作 -> 相机 -> 结果 -> 管理 -> 记录”的顺序，让录入和识别按钮在首屏更早出现。
- 识别结果显示姓名、距离、阈值和解释。
- 管理区只放低频操作，避免干扰主流程。
- 记录区只展示最近关键结果，避免信息过长。
- 继续减少装饰性视觉，保留清晰、克制、答辩友好的风格。

验收：

- 用户不看 README 也能完成录入和识别。
- 老师能从界面直接看到“离线、本地、人脸库、阈值、记录”。
- 所有按钮文字都能说明动作结果。

## 阶段 3：报告与答辩材料

目标：让项目不只是能跑，还能讲得完整。

计划：

- 在报告中补充系统架构图、流程图、测试表。
- 保存真实运行截图：录入、识别成功、未知人员、无人脸、多人人脸、断网识别。
- 明确说明开源项目引用和 Apache-2.0 许可。
- 说明不做活体检测的原因，并把它放入后续改进。

验收：

- 报告结构与 App 实际功能一致。
- 演示清单可以逐项打勾。
- 开源引用完整，避免被认为未注明来源。

## 阶段 4：算法与安全增强

目标：在主流程稳定后再考虑加分功能。

候选方向：

- 人脸质量检测：提示光线差、过曝、模糊、脸太小、贴边、角度偏大。
- 阈值调试页：展示不同阈值对识别结果的影响。
- 视频多人识别稳定化：跨帧跟踪、结果投票、减少姓名跳变。
- 活体检测：加入静态活体或动作活体，但只作为增强功能。
- 数据导出：导出识别记录用于报告截图或测试统计。

取舍：

- 活体检测展示效果强，但容易增加误判和集成成本。
- 阈值调试页更适合写报告，能体现算法理解。
- 人脸质量检测对实际演示最有帮助。

本轮第四阶段已完成低风险落地：

- 已加入 `AdvancedReadinessBuilder`，在测试摘要和完整报告中输出活体检测、模型升级、性能 delegate 和正式发布前置条件。
- 活体检测仍标为未实现，不把照片翻拍防护写成已完成能力；报告会提示先补齐成功识别、未知人员和真机证据后再调研。
- 模型升级不直接替换 FaceNet；只有样本区分清楚时才建议进入 A/B 评测准备，并要求重新统计同人最大 L2、跨人最近 L2、未知人员 L2 和误识/拒识次数。
- 性能 delegate 不直接启用 GPU/NNAPI；只有完成多人视频真机记录后，才建议比较 NNAPI/GPU/XNNPACK 的帧率、发热、黑屏和输出一致性。
- 正式发布边界继续强调 Debug/Release 区分、后端代理云端 Key、SHA-256 记录、隐私授权和 connected 测试。

## 阶段 5：工程化

目标：让项目结构更适合长期维护。

计划：

- 将相机、人脸检测、识别、存储逐步拆成更独立的类。
- 为 `RecognitionEngine` 增加单元测试。
- 为存储文件增加版本号，方便以后迁移。
- 根据需要再考虑 Room，不在主流程稳定前引入。

验收：

- 核心识别逻辑可以独立测试。
- 新功能不会让 `MainActivity` 继续膨胀。
- README、技术设计、报告框架持续更新。

当前进展：

- 已为 `RecognitionEngine` 建立本地单元测试，覆盖空人脸库、精确匹配、最近距离匹配、超过阈值判定未知人员、人员无特征等核心规则。
- 已为 `RecognitionEngine` 加入异常向量防御，维度错误、NaN、Infinity 不会导致识别崩溃。
- 已加入 `FaceQualityAnalyzer`，在录入/识别前阻止脸太小、贴边、侧转过大、倾斜过大的低质量输入，并提供本地单元测试。
- 已加入 `FaceImageQualityAnalyzer`，检查裁剪后人脸图像的亮度和基础清晰度，拦截偏暗、过曝和明显模糊的人脸图像。
- 已加入 `FaceLibraryHealthAnalyzer`，检查本地人脸库是否具备演示条件，并清理维度错误或数值异常的人脸特征向量。
- 已加入同人样本一致性评估，通过组内最大 L2 距离发现录入样本差异过大的人员。
- 已加入跨人员区分度评估，通过最近跨人 L2 距离发现容易混淆的人员组合。
- 已加入阈值校准诊断，根据同人最大 L2 与跨人最近 L2 生成建议阈值或补录建议。
- 已加入答辩验收覆盖分析，测试摘要会提示识别成功、未知人员、边界/拒识、候选人过近和视频多人识别场景是否已覆盖。
- 已把多人视频验收收紧为“至少两张人脸 + 有效稳定身份标签”，避免一张脸或失败标签误算作多人演示完成。
- 已在测试摘要中加入本地离线证明，明确模型、人脸库和识别流程均在本机，不调用云 API。
- 已加入姓名输入规范化，拦截空白、超长和系统保留名称，避免污染人脸库。
- 已加强异常特征防御，录入、识别和人脸库健康检查都会拦截或清理 NaN/Infinity 等异常向量。
- 已抽出 `FaceEmbeddingGuard`，让特征有效性检查、L2 距离计算和同名补录离群判断离开 `MainActivity`，并用单元测试覆盖异常向量和离群样本拒绝。
- 已新增云端 API 模式，可选择 Face++ 托管云端 API 或自部署 CompreFace REST 服务，和本地离线模式组成双引擎识别架构；云端模式提供连接测试按钮，并保存最近一次测试时间、提供商和成功/失败原因，用于提前验证服务地址和 API Key。
- 已加入 `TestSummaryBuilder` 和界面入口，可根据识别记录生成测试摘要，并支持复制摘要文字到报告。
- 已加入 `DemoGuideBuilder` 和首页“下一步”提示，根据人脸库与识别记录自动提示录入、补录、识别成功、未知人员、多人视频和生成摘要等下一步动作。
- 已加入 `FullReportBuilder`，让“复制完整报告”真正包含截图摘要、技术详情、人脸库明细、完整识别记录、云端 API 演示材料、隐私与风险说明和报告使用建议。
- 已加入 `CloudDemoMaterialBuilder`，用于说明 Face++ 免费 Key、CompreFace 自部署、本地离线模式和 GPT 多模态模型的取舍。
- 已抽出 `RecognitionRecordFormatter`，让识别记录显示格式可以单独测试，减少 `MainActivity` 的展示逻辑。
- 已抽出 `RecognitionRecordManager`，让识别记录创建、置顶和最多保留 30 条的规则离开 `MainActivity`，并用单元测试覆盖。
- 已抽出 `CloudRecognitionRouter` 和 `CloudFaceGateway`，让云端提供商分发逻辑离开 `MainActivity`，并用单元测试覆盖 Face++ / CompreFace 路由选择。
- 已加入 `EnrollmentAdvisor`，录入成功后会提示是否建议继续补录到推荐样本数，提高识别稳定性。
- 已把主操作区前移到相机上方，并压缩首屏文案，解决首次打开时核心按钮不够显眼的问题。
- 已增强未知人员解释：当最小距离超过阈值时，结果会显示最接近的已录入人员，方便说明系统为什么判定为未知。
- 已增强识别引擎：加入安全区/稳定区/边界区/拒识区解释、候选人过近拒绝确认、多样本稳定性评分，降低误识别风险。
- 已加入视频多人识别模式：通过 CameraX ImageAnalysis 持续分析预览帧，支持同一画面多张人脸逐张识别，并在预览上叠加人脸框和姓名/未知标签。
- 已为视频多人识别增加短窗口投票和平滑显示，单帧噪声不会立刻替换稳定姓名，新身份需要获得足够票数后才切换。
- 已为视频多人识别增加紧凑调试信息，视频演示记录会保留轨迹号、匹配方式、投票数、确认阈值和 L2 距离，便于解释跨帧确认和排查跳标。
- 已升级测试摘要：统计成功平均距离、边界匹配、拒识区结果和候选人过近拒绝确认次数，便于写实验结果。
- 已加入算法样本评测说明和真机证据链清单，完整报告会明确同人/跨人距离、阈值建议、风险控制信号和必须补齐的真实设备证据。
- 已加入 `ProfileRepository` 和 `RecordRepository`，让人脸库和识别记录的保存、清空、删除与上限裁剪从 `MainActivity` 中收敛到仓储边界。
- 已加入 `DemoStateSnapshot`，为首页下一步提示等 UI 刷新提供轻量状态快照，减少人员、记录、模式和实时状态继续散落。
- 已加入第三阶段小屏保护：360dp 左右窄屏会收紧首页标题、材料按钮、三枚核心操作按钮和全屏多人视频底部面板，降低按钮/长文案/结果区互相遮挡的风险。
- 已收紧发布边界：Debug 包继续允许课程演示所需的本地 HTTP 云端对比，Release 包关闭全局明文 HTTP，并开启资源压缩和代码混淆。
- 已补最小 Android 启动级仪器测试，覆盖 `MainActivity` 打开后核心答辩控件可见；真机摄像头、小屏截图和多人视频录屏仍需设备验收。
- 已完成第四阶段低风险增强边界：测试摘要和完整报告会说明活体、模型升级、性能 delegate、正式发布各自的前置证据和风险，不把未验证能力误写成已实现。
- 后续工程化重点应继续拆分 `MainActivity`，当前已经抽出 `CameraController`、`MotionDirector`、本地/云端/多人视频协调器、报告段落生成器和轻量 Repository；下一步优先抽单人拍照检测流程、状态刷新渲染器和更完整的不可变 UI state。

- 优化人员下拉框空状态，让用户知道还没有录入人员。

2. 基于真机样本记录同人最大 L2、跨人最近 L2、未知人员 L2、光照/角度条件和误识/拒识次数，用于决定是否微调质量阈值。
3. 继续拆分 `MainActivity` 中的拍照检测和状态刷新渲染入口，并在稳定后再评估是否从轻量 Repository 升级到 Room。
